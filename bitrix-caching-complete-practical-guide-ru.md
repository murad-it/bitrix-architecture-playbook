# Кеширование в 1C-Битрикс на продакшене: понятно, по делу и без мифов
### Практическое руководство: как реально ускорять проект и не ловить боль после релизов

> Формат: проблема -> ограничение -> решение -> код -> проверка  
> Цель: собрать понятный и рабочий стандарт кеширования для Bitrix-проекта  
> Для кого: middle/senior разработчики, техлиды, архитекторы, технические владельцы Bitrix-проектов

---

## Почему с кешированием так часто промахиваются

В Bitrix нет одного «кеша». Есть несколько слоев с разной зоной ответственности:

1. Кеш компонента (`StartResultCache`) — кеширует `$arResult` и HTML шаблона.
2. Data cache (`CPHPCache`, `Bitrix\Main\Data\Cache`) — кеширует произвольные данные.
3. Managed cache — централизованный контур хранения/инвалидации данных.
4. Tagged cache — точечный сброс кеша по бизнес-тегам (`iblock_id_X` и др.).
5. ORM cache (`getList(['cache' => ...])`) — кеш результата ORM-запросов.
6. HTML/композитный кеш — ускорение финальной отдачи страницы, особенно для анонимов.

Если смешать эти слои в одну кучу, проект либо не ускорится, либо начнет показывать старые данные в самый неподходящий момент.

---

## Теория простыми словами: как работает кеш и где его пределы

Если по-простому, кеширование в продакшене — это обмен:

1. Мы тратим место на хранение, чтобы меньше тратить CPU/DB на повторный расчет.
2. Мы принимаем контролируемый риск неактуальности, чтобы получить стабильное время ответа.
3. Мы переносим часть сложности из «вычисления» в «инвалидацию».

У любого кеша есть две цены:

1. `Cache hit` (попадание) — быстрое чтение, минимальная нагрузка.
2. `Cache miss` (промах) — полный расчет + запись кеша, иногда очень дорогая.

Чтобы кеш приносил пользу, а не проблемы, держим три принципа:

1. Высокий hit rate на горячих сценариях.
2. Предсказуемая инвалидация по событиям.
3. Контроль персонализации, чтобы не утекали чужие данные.

---

## Термины, которые лучше знать до внедрения

1. `Staleness budget` — максимально допустимая неактуальность данных для конкретного сценария.
2. `Consistency model` — модель согласованности: мгновенная, eventual, смешанная по блокам.
3. `Invalidation contract` — формализованный контракт: какое событие какую область кеша сбрасывает.
4. `Hit ratio` — доля попаданий в кеш на горячем трафике.
5. `Warmup strategy` — стратегия прогрева критических страниц после релизов и массового сброса.

Эти параметры лучше зафиксировать в документации команды и проверять на ревью каждой задачи.

## Карта решений: какой кеш где применять

| Сценарий | Инструмент | TTL | Инвалидация | Примечание |
|---|---|---:|---|---|
| Список новостей | Компонентный кеш + тег `iblock_id_N` | 10–60 мин | По тегу инфоблока | Базовый паттерн для контента |
| Каталог, фильтры, разделы | Data\Cache + TaggedCache | 5–30 мин | Теги + события каталога | Точечный сброс критичен |
| Справочники (редко меняются) | Managed cache | 1–24 ч | Событийный clean() | Отлично для «read-mostly» |
| D7 ORM выборки | `getList(['cache'])` | 1–15 мин | ORM-кеш + события | Следить за JOIN и планом запроса |
| Публичные страницы для анонимов | Композит/HTML кеш | 5–60 мин | Автоинвалидация + исключения | Максимальный прирост TTFB |
| Личный кабинет/корзина | Частично или без кеша | Короткий | Персональные ключи/динамические зоны | Нельзя кешировать общим ключом |

## С чего начать: минимальная модель перед внедрением

Перед написанием кода фиксируй для каждого блока:

1. Что именно кешируем: данные, HTML или оба слоя.
2. Для кого кеш: аноним/авторизованный/группа/сайт/язык.
3. Когда данные устаревают: событие и предельная неактуальность.
4. Где храним и как чистим: ключ, директория, тег, обработчик.

TTL задаем не «на глаз», а из двух вещей: как часто меняются данные и сколько неактуальности бизнес готов терпеть.

## 1) Компонентный кеш: рабочая база для продакшена

```php
<?php
if (!defined('B_PROLOG_INCLUDED') || B_PROLOG_INCLUDED !== true) { die(); }

use Bitrix\Main\Loader;

class CatalogListComponent extends CBitrixComponent
{
    public function onPrepareComponentParams($params): array
    {
        $params['CACHE_TIME'] = (int)$params['CACHE_TIME'] > 0 ? (int)$params['CACHE_TIME'] : 3600;
        $params['CACHE_BY_GROUPS'] = $params['CACHE_BY_GROUPS'] === 'N' ? 'N' : 'Y';
        return $params;
    }

    public function executeComponent(): void
    {
        global $USER;

        $additionalCacheId = [];
        if ($this->arParams['CACHE_BY_GROUPS'] === 'Y' && is_object($USER)) {
            $additionalCacheId[] = $USER->GetUserGroupString();
        }

        if ($this->StartResultCache($this->arParams['CACHE_TIME'], $additionalCacheId)) {
            if (!Loader::includeModule('iblock')) {
                $this->AbortResultCache();
                ShowError('Не подключен модуль iblock');
                return;
            }

            $this->arResult['ITEMS'] = $this->loadItems();
            $this->arResult['TOTAL'] = count($this->arResult['ITEMS']);

            if (empty($this->arResult['ITEMS'])) {
                $this->AbortResultCache();
                ShowError('Данные не найдены');
                return;
            }

            // Внешнему коду нужен только TOTAL, остальное не вытаскиваем наружу.
            $this->setResultCacheKeys(['TOTAL']);
            $this->IncludeComponentTemplate();
        }
    }

    private function loadItems(): array
    {
        return [
            ['ID' => 101, 'NAME' => 'Ноутбук A'],
            ['ID' => 102, 'NAME' => 'Ноутбук B'],
        ];
    }
}
```

Важный момент: по умолчанию кешируется весь `$arResult`.  
`setResultCacheKeys()` не включает кеширование, а ограничивает, что оставлять доступным снаружи и не раздувать кеш.

## 2) Старое и новое ядро: от CPHPCache к D7-кешу данных

### Старый подход (CPHPCache)

```php
<?php
$cache = new CPHPCache();
$ttl = 3600;
$cacheId = 'legacy:section:list:v1:' . md5(serialize($filter));
$cacheDir = '/catalog/sections/';

if ($cache->InitCache($ttl, $cacheId, $cacheDir)) {
    $result = $cache->GetVars();
} elseif ($cache->StartDataCache()) {
    $result = getSectionData($filter);
    $cache->EndDataCache($result);
}
```

### Новый подход D7 (Bitrix Main Data Cache)

```php
<?php
use Bitrix\Main\Data\Cache;

$cache = Cache::createInstance();
$ttl = 3600;
$cacheId = 'd7:section:list:v1:' . md5(serialize($filter));
$cacheDir = '/catalog/sections/';

if ($cache->initCache($ttl, $cacheId, $cacheDir)) {
    $result = $cache->getVars();
} elseif ($cache->startDataCache()) {
    $result = getSectionData($filter);

    $invalid = false;
    if ($invalid) {
        $cache->abortDataCache();
    } else {
        $cache->endDataCache($result);
    }
}
```

API отличается, но логика одна: читаем кеш -> при промахе считаем -> записываем -> сбрасываем по правилам.

## 3) Тегированный кеш: точечный сброс вместо «чистим всё»

```php
<?php
use Bitrix\Main\Application;
use Bitrix\Main\Data\Cache;

$cache = Cache::createInstance();
$taggedCache = Application::getInstance()->getTaggedCache();

$ttl = 1800;
$cacheId = 'catalog:landing:v2';
$cacheDir = '/catalog/landing/';

if ($cache->initCache($ttl, $cacheId, $cacheDir)) {
    $viewModel = $cache->getVars();
} elseif ($cache->startDataCache()) {
    $taggedCache->startTagCache($cacheDir);
    $taggedCache->registerTag('iblock_id_17');
    $taggedCache->registerTag('catalog_price_group_1');

    $viewModel = buildCatalogLanding();

    $taggedCache->endTagCache();
    $cache->endDataCache($viewModel);
}
```

При изменении элемента инфоблока или цены очищается только целевой контур кеша.

## 4) Управляемый кеш: хороший выбор для справочников и настроек

```php
<?php
use Bitrix\Main\Application;

$managed = Application::getInstance()->getManagedCache();
$cacheTtl = 86400;
$cacheId = 'dict:delivery:regions:v1';

if ($managed->read($cacheTtl, $cacheId)) {
    $regions = $managed->get($cacheId);
} else {
    $regions = loadDeliveryRegionsFromDb();
    $managed->set($cacheId, $regions);
}

// Очистка при обновлении справочника:
// $managed->clean($cacheId);
```

Оптимально для данных с частым чтением и редкими изменениями.

## 5) ORM-кеш в D7: ускоряет чтение, но не спасает плохой SQL

```php
<?php
use Bitrix\Iblock\ElementTable;

$rows = ElementTable::getList([
    'select' => ['ID', 'NAME', 'IBLOCK_ID', 'ACTIVE_FROM'],
    'filter' => ['=IBLOCK_ID' => 17, '=ACTIVE' => 'Y'],
    'order' => ['ACTIVE_FROM' => 'DESC'],
    'limit' => 50,
    'cache' => [
        'ttl' => 300,
        'cache_joins' => true,
    ],
])->fetchAll();
```

Ключевые ограничения:

1. ORM cache не лечит плохой SQL.
2. Сначала минимизируй `select` и фильтры, потом кешируй.
3. Для тяжелых JOIN проверяй `EXPLAIN`.

## 6) Сброс по событиям для HL-блоков: практический шаблон

```php
<?php
use Bitrix\Main\EventManager;
use Bitrix\Main\Loader;
use Bitrix\Highloadblock\HighloadBlockTable;

Loader::includeModule('highloadblock');
$eventManager = EventManager::getInstance();

$rs = HighloadBlockTable::getList([
    'select' => ['NAME'],
    'cache' => ['ttl' => 86400, 'cache_joins' => true],
]);

while ($hl = $rs->fetch()) {
    if (!empty($hl['NAME'])) {
        $eventManager->addEventHandler('', $hl['NAME'] . 'OnAfterAdd', 'cleanCacheHighloadBlock');
        $eventManager->addEventHandler('', $hl['NAME'] . 'OnAfterUpdate', 'cleanCacheHighloadBlock');
        $eventManager->addEventHandler('', $hl['NAME'] . 'OnAfterDelete', 'cleanCacheHighloadBlock');
    }
}
```

Практика: обработчик `cleanCacheHighloadBlock` должен очищать только целевые ключи/теги, без глобального `clear cache`.

## 7) Композитный режим: где он дает максимальный эффект

Композит ускоряет первую отдачу HTML для анонимов.  
Подходит для витрин, контентных страниц, листингов.

Не забывай о динамических областях:

1. корзина,
2. авторизация/профиль,
3. персональные цены/скидки,
4. блоки, зависящие от сессии.

Если динамическую область ошибочно оставить статичной, пользователь получает некорректный персональный контент.

## 8) Архитектура cacheId: схема ключей, которая выдерживает рост проекта

Плохой ключ:

```text
catalog_list
```

Хороший ключ:

```text
catalog:list:v3:site=s1:lang=ru:group=2|6:section=17:page=3:sort=price_asc
```

Правило:

1. Версионируй ключ (`v1`, `v2`, `v3`), чтобы безопасно менять формат кеша.
2. Учитывай контекст, влияющий на результат.
3. Не пихай в ключ случайные поля, только реально влияющие.

## 9) Частые ошибки на продакшене и как их чинить

1. Кешируется персональный блок общим ключом.  
   Исправление: добавить в cacheId группу/пользователя или вынести в динамическую область.

2. Большой TTL без событийной инвалидации.  
   Исправление: привязать сброс к `OnAfterAdd/Update/Delete` и тегам.

3. В `$arResult` кешируется избыточный объем данных.  
   Исправление: ограничить состав данных, `setResultCacheKeys()` использовать осознанно.

4. Массовая очистка кеша после любой правки контента.  
   Исправление: точечный `ClearByTag`, `clean($cacheId)`, локальные cacheDir.

5. Нет измеримой проверки результата.  
   Исправление: замерять `TTFB`, SQL count, P95 server time до/после.

---

## 9.1) Критичные пробелы, которые часто пропускают

1. Нет разделения кеша по контурам данных: контент, цены, остатки, персонализация.
2. Нет формального контракта инвалидации по событиям.
3. Не учтены race condition при одновременном прогреве после истечения TTL.
4. Не описана стратегия работы при каскадном сбросе кеша в пиковый трафик.
5. Не учтены особенности многосайтовости, языков и групп доступа в cacheId.
6. Не описаны эксплуатационные метрики и пороговые алерты.

Если хотя бы один пункт отсутствует, кеширование остается неуправляемым.

---

## 9.2) Матрица инвалидации: что и по какому событию очищать

| Домен данных | Событие | Что очищать | Механизм |
|---|---|---|---|
| Инфоблок (элементы/разделы) | `OnAfterIBlockElementAdd/Update/Delete` | Листинги, карточки, связанные блоки | `TaggedCache` + `iblock_id_N` |
| Цены | Изменение цены/типа цены | Каталог, карточка, блоки рекомендаций | Тег цен + targeted cacheId |
| Остатки | Обновление складских остатков | Наличие в карточке/листинге | Короткий TTL + событийный сброс |
| HL-справочники | `OnAfterAdd/Update/Delete` HL | Кеш справочников, lookup-блоки | `ManagedCache::clean` / `Data\Cache` |
| Пользовательские права | Изменение групп/ролей | Блоки с доступом/персонализацией | Пересчет cacheId по группам |

## 9.3) Защита от cache stampede (одновременный пересчет)

Проблема: при истечении TTL много параллельных запросов одновременно запускают тяжелый пересчет.

Практические меры:

1. Использовать «раннее обновление» (soft TTL + hard TTL), где часть запросов продолжает читать старое значение, пока один поток пересчитывает новое.
2. Добавлять jitter к TTL (например, `TTL ± 5-10%`), чтобы избежать синхронного истечения тысяч ключей.
3. Для критичных блоков использовать фоновый прогрев по расписанию.

## 9.4) Персонализация и безопасность кеша

ВНИМАНИЕ: главный риск кеширования в e-commerce и B2B — утечка персональных данных.

Нельзя кешировать общим ключом:

1. персональные цены и скидки,
2. содержимое корзины,
3. блоки с правами доступа,
4. приватные уведомления и персональные статусы.

Минимальное правило безопасности:

1. либо включать контекст пользователя/групп/сайта/языка в cacheId,
2. либо выводить блок через динамическую область (композит),
3. либо отключать кеш для персональных фрагментов.

## 9.5) Эксплуатация: какие метрики считать обязательными

Без метрик кеширование неуправляемо. Минимальный набор:

1. `Cache hit ratio` по ключевым страницам и API.
2. `P50/P95/P99` времени ответа.
3. SQL QPS и количество тяжелых запросов после релиза.
4. Количество событий инвалидации и их распределение по доменам.
5. Доля ошибок/таймаутов при прогреве.

Рекомендуемые сигналы для алертов:

1. падение hit ratio ниже целевого порога,
2. резкий рост miss и SQL-нагрузки,
3. длительный хвост P95/P99 после массовой инвалидации.

## 9.6) Redis/Memcached и файловый кеш: что и когда выбирать

Файловый кеш:

1. простой запуск, минимальная операционная сложность;
2. нормален для небольших/средних нагрузок;
3. хуже работает при высоком I/O и большом количестве мелких ключей.

Redis/Memcached:

1. ниже latency на чтении/записи;
2. лучше масштабируется под высокую конкуренцию;
3. требует контроля памяти, политики вытеснения и мониторинга.

Практический выбор:

1. старт: файловый кеш + корректная инвалидация;
2. рост нагрузки: перенос hot-контуров в Redis/Memcached;
3. highload: обязательный мониторинг hit ratio + memory pressure.

## 10) План внедрения на 1–2 спринта: от диагностики до стабильной работы

1. Выбрать 5 самых медленных страниц по метрикам.
2. Для каждой составить карту данных и источников (DB/API/компоненты).
3. Внедрить компонентный кеш и Data\Cache на тяжелых блоках.
4. Привязать tagged invalidation к изменяемым сущностям.
5. Включить ORM cache на стабильно читаемых выборках.
6. Проверить композит и динамические зоны.
7. Снять повторные замеры и зафиксировать регламент.

Целевые KPI:

1. Снижение SQL-нагрузки на горячих страницах на 30–70%.
2. Стабилизация TTFB и P95.
3. Меньше аварийных `clear cache` и ручного прогрева.

## 10.1) Что делать при инциденте: после релиза сайт резко замедлился

1. Зафиксировать окно инцидента и baseline «до/после».
2. Проверить hit ratio и динамику miss по критическим ключам.
3. Найти каскадные инвалидации и отключить избыточные обработчики сброса.
4. Временно сократить объем рендеринга (опциональные блоки, тяжелые агрегации).
5. Запустить контролируемый прогрев приоритетных страниц.
6. После стабилизации оформить RCA: причина, корректирующие и профилактические меры.

## 10.2) Командный стандарт: Definition of Done для задач с кешированием

Задача считается завершенной только если:

1. определен слой кеширования и контекст cacheId;
2. описан контракт инвалидации;
3. реализованы тестовые сценарии актуальности данных;
4. зафиксированы метрики до/после;
5. добавлен пункт в эксплуатационную документацию.

## 10.3) End-to-end кейс: каталог (листинг + карточка + цена + остаток + персонализация)

Цель кейса: показать рабочую схему, которая закрывает типичный e-commerce сценарий без утечек и с контролируемой актуальностью.

Архитектура кеша по слоям:

1. Листинг каталога (анонимы): компонентный кеш + тег `iblock_id_N`, TTL 10–15 минут.
2. Карточка товара: `Data\Cache` по `PRODUCT_ID`, TTL 5–10 минут, тег инфоблока.
3. Цены и остатки: короткий TTL 60–180 секунд + событийный сброс.
4. Персональные цены: отдельный cacheId с контекстом групп пользователя или динамическая область композита.
5. Корзина и приватные блоки: без общего кеша, только персональный контур.

Контракт инвалидации:

1. Изменение товара: сброс `iblock_id_N` + карточка `product:{ID}`.
2. Изменение цены: сброс ключей цен и блоков, где цена участвует в сортировке/фильтрах.
3. Изменение остатков: сброс блоков наличия, не трогая весь листинг.
4. Изменение прав/групп: инвалидировать персональные ключи или пересобрать контекстный cacheId.

Минимальный формат ключей:

```text
catalog:list:v4:site=s1:lang=ru:section=17:sort=price_asc:page=3
product:card:v3:site=s1:lang=ru:id=1542
product:price:v2:site=s1:group=2|6:id=1542
product:stock:v1:site=s1:id=1542
```

## 10.4) Anti-stampede: рабочий шаблон soft TTL + lock

Ниже практический пример для критичного блока, где недопустим каскадный пересчет при массовом истечении TTL.

```php
<?php
use Bitrix\Main\Data\Cache;

/**
 * Шаблон защиты от stampede:
 * - hardTtl: срок полной валидности кеша
 * - softTtl: после него значение считается устаревающим, но может временно отдаваться
 * - lock: только один поток пересчитывает тяжелые данные
 */
function getCatalogBlockSafe(array $filter): array
{
    $cache = Cache::createInstance();
    $cacheDir = '/catalog/heavy-block/';
    $cacheId = 'catalog:block:v2:' . md5(serialize($filter));

    $hardTtl = 900; // 15 минут
    $softTtl = 300; // после 5 минут пытаемся обновить
    $now = time();

    if ($cache->initCache($hardTtl, $cacheId, $cacheDir)) {
        $payload = $cache->getVars();
        $data = $payload['data'] ?? [];
        $createdAt = (int)($payload['created_at'] ?? 0);
        $lockPath = $_SERVER['DOCUMENT_ROOT'] . '/bitrix/cache/locks/' . md5($cacheId) . '.lock';

        // Если soft TTL не истек, возвращаем кеш без пересчета.
        if (($now - $createdAt) < $softTtl) {
            return $data;
        }

        // Пытаемся захватить lock для фонового пересчета.
        if (!is_dir(dirname($lockPath))) {
            mkdir(dirname($lockPath), 0775, true);
        }

        $lockFp = @fopen($lockPath, 'c');
        if ($lockFp && flock($lockFp, LOCK_EX | LOCK_NB)) {
            try {
                $fresh = buildHeavyCatalogData($filter);
                if ($cache->startDataCache()) {
                    $cache->endDataCache([
                        'data' => $fresh,
                        'created_at' => $now,
                    ]);
                }
                flock($lockFp, LOCK_UN);
                fclose($lockFp);
                return $fresh;
            } catch (\Throwable $e) {
                flock($lockFp, LOCK_UN);
                fclose($lockFp);
                // При ошибке пересчета безопасно возвращаем старый кеш.
                return $data;
            }
        }

        if ($lockFp) {
            fclose($lockFp);
        }

        // Другой поток уже пересчитывает; отдаем предыдущую версию.
        return $data;
    }

    // Cold start: кеша нет, считаем и сохраняем.
    $computed = buildHeavyCatalogData($filter);
    if ($cache->startDataCache()) {
        $cache->endDataCache([
            'data' => $computed,
            'created_at' => $now,
        ]);
    }

    return $computed;
}
```

Примечание: для multi-node инфраструктуры lock лучше переносить в распределенный механизм (Redis lock/DB advisory lock), а не файл.

## 10.5) Нагрузочная валидация кеша: обязательный протокол

Перед релизом проверяем три режима:

1. `Cold cache`: первый прогон после очистки.
2. `Warm cache`: повторный прогон по тем же URL/параметрам.
3. `Invalidate + warmup`: серия изменений данных + контролируемый прогрев.

Минимальные KPI для acceptance:

1. `Warm P95` лучше `Cold P95` минимум на 40%.
2. SQL-запросов на warm-сценарии меньше минимум на 50%.
3. После инвалидации система возвращается к целевому hit ratio за допустимое окно (например, до 5–15 минут).
4. Нет роста 5xx/таймаутов при одновременных запросах в момент истечения TTL.

Минимальный тест-профиль:

1. 3–5 ключевых страниц (главная категория, подкатегория, карточка, поиск, корзина).
2. 2 типа трафика: анонимный и авторизованный.
3. Отдельные сценарии для персональных блоков (цены, скидки, доступы).

Результат теста фиксируется в задаче: до/после по `P95`, SQL count, hit ratio, error rate.

## 11) Чеклист для code review: как понять, что кеш сделан качественно

1. Явно определен слой кеширования (компонент/данные/ORM/композит)?
2. Кеш-ключ учитывает контекст результата?
3. Есть рабочая инвалидация по событиям/тегам?
4. TTL обоснован бизнес-неактуальностью?
5. Измерили метрики до/после?
6. Нет риска утечки персональных данных через общий кеш?

Если минимум по двум пунктам ответ «нет», задача не готова к продакшену.

## 11.1) Чеклист тестирования кеша перед выкладкой

1. Первый запрос (cold cache) отрабатывает корректно и без ошибок.
2. Повторный запрос (warm cache) ускоряется ожидаемо.
3. После события изменения данных показывается актуальный результат.
4. Для разных групп/сайтов/языков не возникает смешивания контента.
5. Персональные блоки не отдаются из общего кеша.
6. Массовая инвалидация не приводит к недопустимой просадке P95.

## Итог

Быстрый Bitrix строится не на галочке «кеш включен», а на системе:

1. Правильный слой кеширования под конкретный сценарий.
2. Точная инвалидация по бизнес-событиям.
3. Контекстные cacheId без утечек и коллизий.
4. Верификация цифрами, а не ощущениями.

Эта связка дает то, что всем нужно на боевом проекте: стабильную скорость, актуальные данные и спокойные релизы без хаотичных сбросов кеша.

Используй эту статью как командный стандарт: архитектура кеша, правила инвалидации, метрики эксплуатации и протокол инцидентов в одном месте.
