# Производительность 1C-Битрикс на production: 10 системных ошибок, которые тормозят проект
### Практический гайд для техлида и backend-команды: архитектура исполнения, кэш, компоненты, интеграции, инфраструктура, контроль регрессий

> Формат: симптом -> причина -> инженерное решение -> проверка результата  
> Цель: ускорять Bitrix-проект системно, а не точечными «костылями»  
> Для кого: middle/senior разработчики, техлиды, архитекторы, технические владельцы продукта

## Что это за статья и чем она отличается от ORM-гайда

Эта статья про **общую производительность 1C-Битрикс-системы**: жизненный цикл запроса, компоненты, кэш, события, интеграции, сеть и контроль регрессий.  
Здесь мы не уходим в глубину D7 ORM (для этого есть отдельный материал).

Если сказать коротко:

- эта статья отвечает на вопрос: **«почему тормозит весь проект?»**
- ORM-статья отвечает на вопрос: **«как писать быстрые запросы и runtime relations?»**

## Карта производительности Bitrix: где обычно теряются миллисекунды

На боевом проекте latency чаще всего распределяется по 6 зонам:

1. База данных (дорогие запросы, много запросов, плохие планы).
2. PHP-слой (лишняя работа в циклах, дубли вычислений, тяжёлые преобразования).
3. Компонентная сборка страницы (много блоков, каскадные вызовы, лишние данные).
4. Кэширование (либо отсутствует, либо инвалидация разрушена).
5. Внешние интеграции (блокирующие API, таймауты, повторы без стратегии).
6. Инфраструктурный контур (PHP-FPM, OPcache, веб-сервер, сеть, CDN).

Главный принцип: не лечим всё подряд. Находим узкое место, устраняем, проверяем цифрами.

## Базовый протокол перед изменениями

Перед оптимизацией фиксируем baseline на одном и том же сценарии:

1. `P95/P99 server time`.
2. Количество SQL-запросов.
3. TTFB и общее время ответа.
4. Peak memory.
5. Количество внешних HTTP-вызовов.
6. Долю времени: БД / PHP / сеть.

Если baseline нет, команда спорит ощущениями, а не фактами.

## 1) D7-контроллер как «комбайн»: всё в одном месте

### Симптом

D7-контроллер или action-обработчик одновременно делает выборки, бизнес-правила, интеграции и подготовку ответа.

### Почему это деградация

- растёт время ответа и связность;
- сложнее кэшировать осмысленно;
- любое изменение бьёт по нескольким слоям сразу.

### Рабочее решение

Разделить слой сценария и слой представления:

1. Сервис или обработчик готовит данные.
2. Контроллер/endpoint вызывает сервис.
3. Presentation-слой только рендерит или сериализует ответ.

```php
<?php
final class CatalogPageService
{
    public function buildViewModel(int $sectionId, int $page): array
    {
        // 1) чтение данных
        // 2) бизнес-правила
        // 3) готовый DTO для UI
        return [];
    }
}
```

### Проверка

Снижение времени endpoint/action, уменьшение сложности ревью, меньше каскадных регрессий.

## 2) Дубли бизнес-логики между компонентами

### Симптом

Одинаковые вычисления цен, доступности, флагов и статусов в нескольких компонентах.

### Почему это боль

- лишняя CPU-нагрузка;
- рассинхронизация результата между страницами;
- сложно кэшировать единообразно.

### Рабочее решение

Вынести правила в единый read-model/service слой и переиспользовать.

## 3) Кэш без модели инвалидации

### Симптом

После изменения контента часть страниц показывает старые данные.

### Причина

Кэш есть, но не привязан к бизнес-событиям изменения данных.

### Рабочее решение

- Проектируем инвалидацию как часть архитектуры, а не как «потом добавим».
- Для контентных сущностей применяем tagged cache.
- TTL не заменяет инвалидацию, а дополняет её.

```php
<?php
use Bitrix\Main\Application;
use Bitrix\Main\Data\Cache;

$cache = Cache::createInstance();
$cachePath = '/catalog/list/';
$cacheId = 'catalog:' . md5($sectionId . '|' . $page);

if ($cache->InitCache(900, $cacheId, $cachePath)) {
    $viewModel = $cache->GetVars();
} elseif ($cache->StartDataCache()) {
    $tagged = Application::getInstance()->getTaggedCache();
    $tagged->startTagCache($cachePath);
    $tagged->registerTag('iblock_id_' . $iblockId);

    $viewModel = []; // сборка данных

    $tagged->endTagCache();
    $cache->EndDataCache($viewModel);
}
```

### Проверка

Нет устаревших данных после изменения контента + стабильное время ответа на повторных запросах.

## 4) «Плоская» пагинация без продуктовой стратегии

### Симптом

На страницу выводятся слишком большие наборы данных.

### Почему это плохо

- растёт TTFB;
- растёт вес HTML/JSON;
- фронтенд начинает тормозить на рендере.

### Рабочее решение

- Ограничивать page size по типу страницы.
- Вводить стабильную сортировку.
- Разделять «витринный» сценарий и «служебный» (админский).

## 5) Событийная система без контроля стоимости

### Симптом

`OnBefore*` и `OnAfter*` обрастают тяжёлыми побочными операциями.

### Почему это бьёт по скорости

Событие срабатывает часто, и даже «маленькая» логика становится дорогой в сумме.

### Рабочее решение

1. В `OnBefore*` оставлять только лёгкую валидацию.
2. Тяжёлые действия выносить в фон.
3. Проводить регулярный аудит обработчиков.

```bash
rg "AddEventHandler|EventManager::getInstance\(\)->addEventHandler" /path/to/project
```

## 6) Внешние API в синхронном рендере без плана деградации

### Симптом

Сторонний сервис «подвис» — страница тормозит или падает.

### Рабочее решение

- Таймауты и контроль статусов обязательны.
- Дедупликация одинаковых запросов в рамках одного хита.
- Резервный сценарий (кэшированное значение или упрощённый UI).
- Тяжёлые массовые задачи выносить в async-контур.

```php
<?php
$http = new \Bitrix\Main\Web\HttpClient([
    'socketTimeout' => 2,
    'streamTimeout' => 3,
]);

$response = $http->get($url);
$status = $http->getStatus();

if ($status !== 200 || $response === false) {
    // fallback сценарий
}
```

## 7) Нет локального кэша на время запроса

### Симптом

В рамках одного HTTP-запроса повторяются одинаковые вычисления/получения данных.

### Рабочее решение

Использовать лёгкий кэш в памяти одного запроса, не смешивая его с долговременным кэшем.

```php
<?php
final class RequestCache
{
    private array $storage = [];

    public function remember(string $key, callable $callback): mixed
    {
        if (!array_key_exists($key, $this->storage)) {
            $this->storage[$key] = $callback();
        }
        return $this->storage[$key];
    }
}
```

## 8) Фронтенд-перегруз на стороне Bitrix-страницы

### Симптом

Сервер отдаёт страницу быстро, но пользователь видит медленный интерактив.

### Причина

Лишние JS-бандлы, ненужные библиотеки, тяжелые инлайны, отсутствие критического разделения.

### Рабочее решение

- Отключать неиспользуемые JS/CSS зависимости.
- Контролировать вес и последовательность подключения.
- Измерять не только backend метрики, но и Web Vitals.

---

## 9) Инфраструктура «по умолчанию», без профиля нагрузки

### Симптом

Код уже оптимизирован, а latency всё равно нестабильный.

### Что проверить

1. Настройки PHP-FPM (workers, max children).
2. OPcache и реальные hit rates.
3. Keep-alive и настройки веб-сервера.
4. CDN/кэш статики.
5. Логи 499/502/504 и пики по времени.

Без инфраструктурного профиля часть оптимизаций в коде не даст эффекта.

---

## 10) Нет технического регламента по производительности

### Симптом

После каждой новой фичи производительность плавно деградирует.

### Рабочее решение

Ввести лёгкий регламент перед `merge` и `release`:

1. Обязательные замеры на критичных сценариях.
2. Check-list по кэшу, событиям, интеграциям.
3. Перф-отчёт «до/после» для рискованных изменений.

---

## Пример отчёта «до/после» для техлида

| Метрика | До | После |
|---|---:|---:|
| SQL-запросы | 328 | 77 |
| P95 server time, ms | 2150 | 640 |
| Peak memory, MB | 32 | 15 |
| Внешние HTTP-вызовы | 12 | 4 |

Что помогло в этом кейсе:

- разделение компонентной логики;
- корректная кэш-стратегия с инвалидацией;
- чистка событий и вынос тяжёлых операций в фон;
- fallback-логика для внешнего API;
- контроль frontend-веса страницы.

---

## Что проверить перед merge в main

1. Есть baseline и целевые метрики.
2. Изменение устраняет конкретный bottleneck, а не «всё сразу».
3. Кэш связан с инвалидацией, а не только с TTL.
4. Внешние API защищены таймаутами и fallback.
5. События не содержат тяжёлых побочных эффектов.
6. Есть проверка влияния на UX и Web Vitals.
7. Есть повторный замер после внедрения.

---

## Итог

Производительность Bitrix-проекта почти никогда не лечится одной «умной правкой».  
Рабочий результат появляется там, где команда мыслит системно: разделяет ответственность слоёв, держит в порядке кэш и интеграции, меряет эффект цифрами и не выпускает изменения без технической проверки.
